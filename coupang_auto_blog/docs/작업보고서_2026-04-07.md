# 작업보고서 2026-04-07

## 작업 내용: 사이트맵 자동 반영 복구 및 원인 정리

---

## 배경

- 사이트맵 자동 반영 기능은 이전에는 정상 동작했던 것으로 보임
- 2026-04-07에 우연히 `sitemap.xml` 상태를 확인하는 과정에서 이상 발견
- 외부 공개 사이트맵에서는 상세 URL이 사라졌고, 서버 내부 `http://127.0.0.1:3000/sitemap.xml` 응답은 완전히 빈 `<urlset>` 상태였음

즉, "새 글이 올라오면 사이트맵에 자동 반영된다"는 기대 동작이 깨진 상태였음.

---

## 확인된 증상

### 1. 공개 사이트맵 이상

공개 `sitemap.xml`에는 아래 정적 URL만 노출됨:

- `/`
- `/reviews`
- `/recipes`
- `/news`
- `/search`

리뷰/레시피/뉴스 상세 URL은 전부 누락.

### 2. 내부 사이트맵은 완전 빈 상태

서버 내부 확인 결과:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
</urlset>
```

즉, `web`이 실제 사이트맵 계산 단계에서 빈 배열을 반환하고 있었음.

### 3. automation-server는 정상

아래 엔드포인트들은 모두 정상 응답 확인:

- `GET /api/reviews/sitemap`
- `GET /api/recipes/sitemap`
- `GET /api/news/sitemap`

따라서 DB나 automation-server 문제가 아니라 `web/app/sitemap.ts` 쪽 문제로 범위가 좁혀짐.

## 원인 분석

### 원인 1. 2026-03-23 빌드 최적화의 부작용

기존 빌드 최적화 작업에서 빌드 중 외부 fetch 재시도를 막기 위해 아래 코드가 들어가 있었음.

```typescript
if (process.env.NEXT_PHASE === "phase-production-build") return [];
```

이 로직은 빌드 중 retry를 줄이는 데는 효과가 있었지만, sitemap 자체를 **빈 결과로 고정**시키는 부작용이 있었음.

결과:

- 빌드 시 빈 sitemap이 생성됨
- 그 산출물이 정적으로 서빙됨
- 이후 새 글이 올라와도 상세 URL이 반영되지 않음

### 원인 2. 사이트맵이 너무 무거운 공개 목록 API를 직접 사용

기존 `sitemap.ts`는 다음 경로들을 직접 호출했음:

- `/api/reviews?limit=1000`
- `/api/recipes?limit=1000`
- `/api/news?limit=1000`

이 API들은 목록용 응답이라 본문, 설명, 재료, instructions 등 불필요하게 큰 데이터를 포함함.
사이트맵 생성에는 `slug`, `id`, `updatedAt` 정도만 필요하므로 비효율적이었음.

### 원인 3. 운영 절차 문서 혼선

문서 일부는 일반 코드 수정에도 `docker compose build --no-cache`를 반복 사용하도록 안내하고 있었음.

이 때문에:

- `web`의 `npm install`
- Next.js 프로덕션 빌드
- 정적 생성 단계

가 모두 캐시 없이 반복되어 배포 시간이 과도하게 길어짐.

이번 사이트맵 문제의 직접 원인은 아니지만, 확인/복구 속도를 느리게 만드는 운영상 문제로 확인됨.

## 수정 내용

### 1. automation-server에 사이트맵 전용 경량 엔드포인트 추가

추가된 경로:

- `GET /api/reviews/sitemap`
- `GET /api/recipes/sitemap`
- `GET /api/news/sitemap`

반환 데이터:

- 리뷰: `slug`, `updatedAt`
- 레시피: `id`, `slug`, `updatedAt`
- 뉴스: `id`, `slug`, `updatedAt`

효과:

- 사이트맵 생성 시 불필요한 본문/메타데이터 로드 제거
- 대량 데이터 환경에서 사이트맵 생성 부담 감소

### 2. `web/app/sitemap.ts` 수정

변경 사항:

- 무거운 목록 API 대신 사이트맵 전용 경량 API 사용
- `export const dynamic = "force-dynamic"` 추가
- 빌드 시 `return []` 제거
- 빌드 단계에서는 최소한 정적 페이지 목록은 유지
- 런타임에서는 실제 리뷰/레시피/뉴스 sitemap API를 읽어 상세 URL 생성

핵심 수정 방향:

```typescript
const shouldSkipDynamicFetch = process.env.NEXT_PHASE === "phase-production-build";
if (shouldSkipDynamicFetch) return staticPages;
```

즉, 빌드 시에는 빈 사이트맵이 아니라 정적 기본값만 만들고, 실제 운영 중에는 상세 URL까지 포함한 사이트맵을 계산하도록 변경.

### 3. revalidate 경로에 sitemap 추가

`web/app/api/revalidate/route.ts`에 `/sitemap.xml` 재검증을 추가.

효과:

- 필요 시 수동 재검증으로 사이트맵 갱신 가능
- 운영 중 확인/복구 포인트가 명확해짐

## 기대 동작

이제 새 글이 `published` 상태로 저장되면:

1. automation-server의 sitemap 전용 API에서 해당 글을 반환
2. `sitemap.xml` 요청 시 `web/app/sitemap.ts`가 최신 데이터를 다시 계산
3. 리뷰/레시피/뉴스 상세 URL이 사이트맵에 자동 포함

즉, 별도 수동 수정 없이 "새 글 게시 → 사이트맵 자동 반영" 흐름으로 복구됨.
